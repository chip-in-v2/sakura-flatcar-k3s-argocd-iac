# 理由書
このリポジトリが構築するサイトにおいて、それぞれの技術要素を選択する理由について述べる。

## サーバレス指向の不採用
AWS, Azure, GCP などサーバレスを目指さない理由は以下のとおり。
- Lambda, Fermyon などの FaaS や S3 によるサーバレスソリューションは処理時間、ストレージ、メモリなどの制約が大きく、コードやアーキテクチャに特殊な技術が必要となったり、外部の追加のマネージドサービスに頼ることになったりする
- ログやメトリックスの採取、セキュリティ監視、データベースのマネージドサービスなどは結局24h365であり、サーバレスといってもそれほど安くすることができない
- エンドユーザからの最初のアクセスに対するUXを考えると24h365のサーバがある方が設計しやすい

## AZ非冗長

AZ冗長について、以下の利用でこれを実装しない。
- 構成の複雑度が上がる割にその効果が見えにくい
- 性能上も AZギャップと呼ばれるレイテンシがあることがわかっている

## VPSベンダの不採用

VPSベンダとしてさくらのVPS、Conoha Wing、Vultr、Linode(Akamai Connected Cloud)があるが、ISMAP を取得しているベンダはない。ベアメタルのサービスも同じ理由で採用できない。


## OSとセキュリティ

- OSはセキュリティ面を考慮してイミュータブルな Flatcar Container Linux を使う
- EDRとして eBPF の Tetragonを入れる。
- Packer を使用して Vultr のオフィシャルイメージに nomad, consul, tetragon を追加インストールした[スナップショットを作成](./flatcar.drawio.svg)し、これをベースとしてterraform で構築する

### ウィルススキャン

イミュータブルOS＋コンテナで運用し、eBPF でイミュータブル領域への書き込みを監視するので、それらの領域におけるウィルススキャンは不要である。ユーザがアップロード・ダウンロードするファイルだけを ClamAV などのOSSのツールでスキャンすべきであるが、それはアプリケーションの機能とする。

### WAF

脆弱性をついてくる攻撃に対しては SBOM の管理などで対応するため不要である。DDOS など過負荷攻撃対策については基本的にはアプリケーション領域で対応すべきであるが、 iptables での制御も検討する（未検討）

### eBPF

OSS の eBPF としては、検知の Falco か防御の Tetragon の選択肢があるが、 Falco はユーザ空間で処理を行い、 Tetragon はカーネル内で完結するという処理方式の違いから Tetragon の方が性能負担は低い。したがって Tetragon を選択する。

Tetragon には有償版の Isovalent Enterprise があり、これを導入することで、以下のメリットがあることがわかっているが、価格などの問題もあり、

1. 業界標準に準拠したポリシーセット

自分で「攻撃の手口」を研究してポリシーを書く必要がなく、テンプレートを選択するだけで適用できます。

- **PCI-DSS / SOC2 / HIPAA 準拠パック**: 監査に必要なファイルアクセス監視や特権変更のログ取得設定。
- **MITRE ATT&CK マッピング**: 攻撃者の戦術（難読化、永続化、権限昇格など）を検知するためのベストプラクティス設定。

2. ポリシー生成の自動化 (Policy Generation)

これが技術者にとって最大のメリットかもしれません。

- **学習モード**: 対象のアプリケーションを一定期間動かすと、その挙動（ネットワーク通信、プロセス実行、ファイルアクセス）を自動で学習します。
- **自動生成**: 学習した「正常な挙動」をベースに、それ以外の動きを検知・遮断するための Tetragon TracingPolicy (YAML) を自動生成します。

ホワイトリスト化: 自分で試行錯誤してホワイトリストを作る手間が劇的に削減されます。

1. 特権的なフィードとインテリジェンス

- **CVE 連携**: 新たな脆弱性（CVE）が発見された際、その脆弱性を突く挙動を特定するためのポリシーが Isovalent 社から迅速に配信されます。
- **不審なバイナリ/ドメインの判定**: eBPFでキャッチしたイベントに対して、リアルタイムで脅威インテリジェンスを照合する機能が統合されています。

## クラスタソフト

クラスタソフトには以下の理由で k3s を選択した。

- k8s, k0s はコントロールプレーンが分離されていて複雑度が高く、ホスト間の対称性がないため、VMの構成設計が難しい
- RKE2 も重いのと、自慢のネットワークプラグインが使用できないため採用しない

## コンテナレジストリ

Docker Hub などの公式イメージ（nginx:latest や node:alpine など）をそのままデプロイするのではなく、一旦手元で Pull し、さくらのクラウドのコンテナレジストリに Push してから本番環境等で使用する

### 1. サプライチェーンリスクの低減（イメージの不変性の担保）
パブリックレジストリのタグ（例: latest や 1.0.0）は、提供者が上書きすることが可能です。もし提供者のアカウントが乗っ取られ、悪意のあるコードが仕込まれた同名のイメージで上書きされた場合、直接Pullしていると次回デプロイ時に自動的に汚染されたイメージを取り込んでしまいます。
一度安全を確認したイメージをさくらのクラウドのコンテナレジストリなどのプライベート環境にPushしておくことで、外部の変更に引きずられない「確実な不変性（イミュータビリティ）」を担保できます。

### 2. ダウンタイムや障害からの保護（可用性の向上）
Docker Hub などの外部サービスが障害でダウンしていると、オートスケール時や再デプロイ時にイメージがPullできず、システム障害に直結します。自社のインフラ（Vultr）と同じデータセンター群にあるレジストリを使用することで、外部障害の影響を切り離すことができます。

### 3. 外部レジストリのレートリミット（利用制限）回避
Docker Hub は、無料・匿名ユーザーに対して「6時間あたり100Pullまで」という厳しいレートリミットを設けています。CI/CDパイプラインを頻繁に回したり、複数台のサーバーへ一斉にデプロイしたりすると、すぐに制限に引っかかりデプロイが失敗します。さくらのクラウドのコンテナレジストリを介することで、この制限を気にする必要がなくなります。

### 4. デプロイの高速化と通信コストの削減
さくらのクラウドのコンピュートリソースから、同じさくらのクラウドのコンテナレジストリへアクセスする場合、ネットワーク的に非常に近く、通信帯域も太いため、イメージのPullが高速に完了します。また、外部へのアウトバウンド通信（エグレストラフィック）を減らすことにも繋がります。

### 5. 厳格なアクセス制御（RBAC）
パブリックイメージであっても、自社の運用に組み込む段階で「誰（どのサーバー・どの開発者）がアクセスしてよいか」を Vultr の機能で厳格に制御できます。これにより、意図しない環境への展開や、内部犯行によるイメージの改ざんリスクを抑えることができます。

## ストレージ

基本的にワーカホストのディレクトリをバインドマウントして使用する。冗長化は上位層のDBMSなどにより実装する。これにより、高性能と低価格を両立する。

## インターネットからの攻撃防御

インターネットから無差別に攻撃してくるもののほとんどは IP アドレスをスキャンしてサーバを発見しており、サーバの正規のFQDNへのアクセスとなっていない。
SNI の段階で FQDN が一致しないものについてはハンドシェイクに失敗するように設定する。これにより通信が後段に回ることがなく、攻撃はもちろん検索のほとんどを失敗させることができる。

WAFについては、そのほとんどの防御ルールが関係のないソフトウェアに関するものであり、有効性に疑問がある。使用しているアプリ、ミドルウェアに対する防御ルールが発見されてから導入を検討する。

## ロードバランサ

L7のリバースプロキシの段数をできるだけ少なくしたいので、L7ロードバランサの類は使用せず、L4ロードバランサを使用する

## サーバ証明書

ACMEで自動的に更新できる Let's Encrypt を使用する。ACME DNS-01 プロトコルで cert-manager に取得させる。HTTP-01 を使用せずに DNS-01で取得することで HTTP 未開通のような問題を避ける。

## DNS

Vultr のDNSはサブドメインの委任ができないので、使用せず、Digital Ocean DNS を使用することにした。
external-dns は consul との連携ができないことが判明して、使用するのをやめて、 DigitalOcean の API を直接呼び出して Aレコードを登録することにした。
Traefik の ACME チャレンジにも DigitalOcean プロバイダを使用する。
